# 仓颉语言 AI 编程竞赛题目设计（候选方案）

> 本文档提出 70 个竞赛题目设计方向，涵盖 Web 框架、数据处理、游戏引擎、科学计算、加密编码、物理仿真、开发工具等多个领域。
> 每道题目均独立于当前 CangjieLua 解释器项目，可作为全新仓颉项目进行开发。
> 设计原则：**架构丰富、思维友好、循序渐进、方便测评、趣味实用**。

---

## 设计原则说明

| 原则 | 说明 |
|------|------|
| **架构丰富** | 项目有清晰的模块划分（解析层 / 核心层 / 输出层），代码量可达数百至上千行 |
| **思维友好** | 不依赖编译原理、类型论等深层理论，主要考察数据结构、算法和工程能力 |
| **循序渐进** | 每题拆分为 6-10 个子任务，从基础到进阶，适合渐进式计分 |
| **方便测评** | 每个子任务有明确的输入/输出或断言，可自动化判定正确性 |
| **趣味实用** | 选题贴近真实开发场景，参赛者完成后有成就感和实际收获 |

### 题目 1：Markdown → HTML 转换器

**领域**：文本处理  
**难度**：⭐⭐（中等）  
**简介**：实现一个 Markdown 子集到 HTML 的转换器，输入 Markdown 文本字符串，输出对应的 HTML 字符串。

**考察要点**：字符串逐字符/逐行解析、递归处理行内元素、数据建模

**子任务设计**：

| 子任务 | 分值 | 描述 |
|--------|------|------|
| 1.1 标题转换 | 10 | `# H1` → `<h1>H1</h1>`，支持 1-6 级 |
| 1.2 段落识别 | 10 | 连续非空行合并为 `<p>` 标签，空行分段 |
| 1.3 粗体与斜体 | 15 | `**粗体**` → `<strong>`，`*斜体*` → `<em>` |
| 1.4 行内代码 | 10 | `` `code` `` → `<code>code</code>` |
| 1.5 链接 | 15 | `[text](url)` → `<a href="url">text</a>` |
| 1.6 图片 | 10 | `![alt](src)` → `<img src="src" alt="alt">` |
| 1.7 无序列表 | 15 | `- item` 系列转换为 `<ul><li>` 嵌套 |
| 1.8 有序列表 | 10 | `1. item` 系列转换为 `<ol><li>` 嵌套 |
| 1.9 代码块 | 15 | ` ``` ` 围栏代码块 → `<pre><code>` |
| 1.10 综合转换 | 20 | 混合多种语法的完整文档转换 |

**测评方式**：给定 Markdown 输入字符串，比较函数输出的 HTML 与期望 HTML 是否一致（忽略多余空白）。

**示例**：
```
输入: "# Hello\n\nThis is **bold** and *italic*."
输出: "<h1>Hello</h1>\n<p>This is <strong>bold</strong> and <em>italic</em>.</p>"
```

## 评测框架建议

每道题目建议采用统一的测评模式：

```
// 伪代码示意
func runTestCase(input: TestInput): TestOutput {
    // 调用参赛者实现的核心函数/类
    let result = Solution.solve(input)
    return result
}

func verify(expected: TestOutput, actual: TestOutput): Bool {
    // 精确比较或容差比较
}
```

**通用原则**：

1. **纯函数式接口**：每个子任务通过函数调用评测，避免依赖文件 I/O 或网络
2. **字符串化比较**：复杂结构通过 toString() 序列化后比较
3. **浮点容差**：数值类题目使用 1e-6 容差
4. **集合无序比较**：结果集不要求特定顺序时，排序后比较
5. **增量评测**：前面子任务不通过不影响后续子任务单独评测
